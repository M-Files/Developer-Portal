## Security context object

Every imported object **must** have a `securityContextObject`. This is a reference to an existing Advanced object (e.g. a Project) whose ACL (Access Control List) determines who can access the imported object.

```json
{
  "securityContextObject": {
    "objType": 101,
    "id": 23
  }
}
```

| Field | Type | Description |
|-------|------|-------------|
| `objType` | int | The object type ID of the parent object |
| `id` | int | The ID of the parent object |

The security context links a Basic object to the Advanced object that controls its permissions. When M-Files checks whether a user can access a Basic object, it evaluates the ACL of the referenced security context object.
{:.note}

## How objects relate through security context and tags

Consider a scenario where Object B is an Advanced object (e.g. a Project), and Objects A, C, and D are Basic objects that reference it:

```
Object A ──┐
            ├──► Object B (Advanced / Project)
Object C ──┤     ▲ security context
            │
Object D ──┘
```

When Basic objects are created:

1. Their lookup properties that reference Advanced objects are used to **generate tags**
2. Tags follow the pattern: `related-{vaultId}-{referencedObjectType}` = `{referencedObjectId}`
3. When the Advanced object is viewed, the system searches for all Basic objects tagged with references to its ID

### Tag generation example

Given a Basic object with property 1078 referencing a Project (objType=101, id=23):

* **Generated tag:** `related-{vaultId}-101` = `"23"`
* **Result:** When Project id=23 is opened, this Basic object is found as a related object

This creates an efficient reverse lookup — Advanced objects can find all Basic objects that reference them without scanning every object.

## Property values

Every object has an array of property values. At minimum, three properties are required:

| PropertyDef | Name | Required | Data type | Description |
|-------------|------|----------|-----------|-------------|
| 0 | Name/Title | Yes | `text` | Display name of the object |
| 25 | Created by | Yes | `lookup` | The user who created the object (user ID) |
| 100 | Class | Yes | `lookup` | The class of the object (class ID) |
| 22 | Single file | **Do not send** | — | Auto-calculated from file count |

All other built-in properties (timestamps, version info, GUIDs, etc.) are auto-generated by the server.

Property definition IDs, class IDs, and GUIDs are vault-specific. The values shown in examples must be adjusted to match your vault configuration.
{:.note.warning}

### PropertyValue structure

```json
{
  "propertyDef": 0,
  "value": {
    "dataType": "text",
    "value": {
      "text": "My Document"
    }
  }
}
```

### Supported data types

| dataType | Value field | Example |
|----------|------------|---------|
| `text` | `"text"` | `{ "text": "My Document" }` |
| `integer` | `"integer"` | `{ "integer": 20 }` |
| `number` | `"number"` | `{ "number": 3.14 }` |
| `boolean` | `"boolean"` | `{ "boolean": true }` |
| `timestamp` | `"timestamp"` | `{ "timestamp": "2025-01-15T12:00:00Z" }` |
| `date` | `"date"` | `{ "date": "2025-01-15" }` |
| `time` | `"time"` | `{ "time": "14:30:00" }` |
| `lookup` | `"lookups"` | See [lookup example](#lookup-values) below |
| `multiSelectLookup` | `"lookups"` | See [lookup example](#lookup-values) below |

Only populate the field matching your `dataType`. Other fields can be omitted.

### Lookup values

Lookups reference other objects (classes, users, value list items, etc.):

```json
{
  "propertyDef": 100,
  "value": {
    "dataType": "lookup",
    "value": {
      "lookups": [
        {
          "name": "Other Document",
          "guid": "{00000000-0000-0000-0000-000000000000}",
          "objId": { "objType": 1, "id": 94 }
        }
      ]
    }
  }
}
```

## Object versions

* All objects have versions that track changes over time
* Most operations create new versions rather than modifying existing ones
* The latest version is returned by default on GET requests

## Async processing

Write operations (POST, PUT) are processed asynchronously:

* **JSON-only POST** returns **201 Created** (object metadata is created synchronously)
* **Multipart POST** (with files) returns **202 Accepted** (files are processed asynchronously)
* **PUT** always returns **202 Accepted**

The response includes a `Location` header with the URI of the created/updated resource.

## Error responses

Errors follow the HTTP Problem Details format (RFC 7807):

| Status | Meaning | Action |
|--------|---------|--------|
| 400 | Bad request — malformed payload or missing required fields | Fix the request body |
| 401 | Unauthorized — token expired or invalid | Refresh the access token |
| 403 | Forbidden — insufficient scope | Ensure token has `mfmasterstorage` scope |
| 404 | Not found — object or file does not exist | Verify the IDs in the URL |
| 413 | Payload too large — file exceeds 25 MB | Use the [SAS token flow]({{ site.baseurl }}/APIs/EXTERNAL-API/Large-File-Upload/) |
| 429 | Too many requests — rate limited | Back off; respect `Retry-After` header |
| 500 | Internal server error | Retry with exponential backoff |
